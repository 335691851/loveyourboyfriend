# GitHub README 展示设计

## 目标

将仓库首页 README 重构为产品与技术并重的项目展示页。访客应能在一分钟内理解产品定位、看到真实移动端界面，并快速定位架构、本地启动、测试和部署信息。

## 目标读者

- 想体验产品的普通访客
- 评估项目完成度的开发者或招聘方
- 准备复用、部署或继续开发项目的维护者

## 信息结构

README 采用由产品到技术的叙事漏斗：

1. Hero：项目名、短标语、在线体验和服务状态入口
2. 真实移动端截图与产品定位
3. MVP 核心能力和使用边界
4. Mermaid 系统架构图
5. Mermaid 聊天、长期记忆和语音数据流图
6. LangChain 模块与职责
7. Monorepo 目录结构和技术栈
8. 本地启动、环境变量与 Supabase 初始化
9. API、测试和质量检查
10. Vercel、Render、Supabase 生产部署链路
11. 隐私、安全、90 天保留策略与项目状态

## 视觉内容

- 从正式站点以移动端视口截取一张不包含真实用户历史、密钥或控制台信息的产品图。
- 图片保存到 `docs/assets/` 并使用仓库相对路径，保证 GitHub 可直接渲染。
- 使用 GitHub 原生 Mermaid 展示系统组件关系和核心数据流，避免依赖外部绘图服务。
- 徽章只展示仓库中能够核实的技术和部署信息，不使用虚构的测试覆盖率或性能数字。

## 内容原则

- 中文为主，保留标准英文技术名词。
- 明确“陆川”是虚构 AI 聊天伙伴，不暗示真人身份。
- 区分已实现能力、需要平台配置的能力和后续方向。
- 所有命令、目录、环境变量、模型名称、接口和部署参数必须与当前代码一致。
- 不在 README 中出现 API Key、数据库密码或其他私密配置。
- 不承诺无法由仓库验证的响应时间、无限额度或情感效果。

## 验收标准

- GitHub Markdown、相对链接、代码块和 Mermaid 语法完整。
- 真实截图能从仓库正常加载，且适合移动端产品展示。
- 本地启动、测试、构建和部署命令与项目脚本一致。
- LangChain、匿名身份、RLS、长期记忆、语音和数据清理描述与代码一致。
- README 不包含 `TBD`、`TODO`、占位图片或未经验证的功能声明。
